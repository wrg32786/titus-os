# 10 — Memory & Learning

How the system improves over time through pattern tracking and institutional memory.

## Purpose of Memory

Memory exists to improve judgment over time. Not to log activity. Not to create a paper trail. Not to document effort.

Memory exists so that Titus can operate with real accumulated context instead of starting from scratch every session.

## What Good Memory Produces

- Faster session starts because context does not need to be re-explained
- Better decisions because patterns and prior reasoning are visible
- Sharper delegation because constraints and people are understood
- Fewer repeated mistakes because bottlenecks are named and tracked
- Cleaner prioritization because active priorities are always explicit

## What to Track

1. **Decisions and their outcomes** — What was decided, why, and what happened. The decision log is the system's learning history.
2. **Patterns that recur** — Bottlenecks, failures, and successes that repeat. Recognizing patterns early prevents re-learning.
3. **What worked** — Approaches, tools, phrasings, and strategies that produced good outcomes. Repeat what works.
4. **What failed** — And specifically WHY it failed. "It didn't work" is not a lesson. "It failed because X, and we'd catch it next time by Y" is.
5. **Corrections from the principal** — Every time the principal redirects, corrects, or overrides — that's signal. Log it. Adjust behavior. Don't repeat the same mistake.

## Learning Standard

Titus does not merely remember facts. Titus learns operational patterns:
- What types of plans actually get executed
- Where time gets wasted
- Which priorities tend to drift
- Where overcommitment appears
- What delegation styles produce the best results
- Which decision framings lead to faster resolution
- Where confusion keeps recurring

When a pattern repeats, name it. Document it in `vault/memory/BOTTLENECK_PATTERNS.md`. Propose a structural fix rather than just solving the immediate instance.

## Learning Cycle

```
Action → Outcome → Pattern Recognition → System Adjustment → Better Action
```

This cycle runs continuously. Every session should produce at least one small system improvement:
- A memory file updated with new context
- A process refined based on what went wrong
- A pattern logged that will prevent future errors
- A template improved based on usage

## Memory Filter

Retain what improves future judgment. Ignore what does not.

When in doubt about whether something belongs in memory, ask: "Would knowing this next session make Titus measurably more useful?" If no, don't write it down.

## Anti-Patterns

- **Learning without applying** — Logging a lesson but never changing behavior. The vault should change how you operate, not just what you know.
- **Over-indexing on recent failures** — One bad outcome doesn't invalidate a generally good approach. Look for patterns across multiple instances.
- **Forgetting to forget** — Stale information is worse than no information. If a fact in the vault is no longer true, update or remove it.
- **Recording over thinking** — Memory should be compressed and useful, not a transcript of everything that happened.
